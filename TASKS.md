# План задач по разработке Timetable Bot

## 0) Подготовка проекта
- [ ] Создать структуру репозитория: `api`, `domain`, `application`, `infrastructure`, `docs`.
- [ ] Настроить Spring Boot + Spring WebFlux, систему конфигурации и профили окружений.
- [ ] Подключить MongoDB и RabbitMQ (локальные docker-compose профили для dev/test).
- [ ] Настроить базовый CI: сборка, линтеры, тесты, проверка форматирования.
- [ ] Описать `.env.example` без секретов (токены/ключи только через переменные окружения).

## 1) Авторизация и роли (Admin API)
- [ ] Реализовать endpoint `POST /auth` для получения API-ключа/Bearer-токена администратора.
- [ ] Реализовать middleware/фильтр проверки ключа для административных endpoint.
- [ ] Добавить модель ролей (`USER`, `ADMIN`) и политику доступа.
- [ ] Добавить тесты на успешную/ошибочную авторизацию и доступ без токена.

## 2) API слой Telegram
- [ ] Реализовать обработчик команды `/start` с созданием пользователя при первом обращении.
- [ ] Реализовать `/menu` с актуальным списком пользовательских команд.
- [ ] Реализовать маршрутизацию команд в `MessageHandlerModule`.
- [ ] Добавить единый формат сообщений об ошибках ввода и подсказки по повторному действию.

## 3) Модуль пользователей
- [ ] Реализовать `UserModule` и `UserRepository` для профиля, таймзоны, предпочтений недели.
- [ ] Добавить хранение состояния onboarding/первого запуска.
- [ ] Реализовать защиту от доступа к данным других пользователей.

## 4) Модуль расписания
- [ ] Реализовать сущность события расписания (занятие/экзамен, время, место, источник).
- [ ] Реализовать команды `/today`, `/tomorrow`, `/week`.
- [ ] Реализовать ручное добавление, редактирование и удаление событий.
- [ ] Ограничить количество событий расписания на пользователя (до 5000).
- [ ] Оптимизировать `/week` до целевого SLA (до 2 сек без внешних API).

## 5) Импорт расписания
- [ ] Реализовать `/import_timetable` для загрузки CSV/iCal.
- [ ] Реализовать парсеры CSV и iCal + валидацию строк.
- [ ] Реализовать импорт из внешнего API учебного заведения.
- [ ] Добавить историю импортов и статусы (успешный/частичный/ошибка).
- [ ] Обеспечить правило: не более одного активного импорта на пользователя.
- [ ] Гарантировать, что при неудачном импорте существующее расписание не теряется.

## 6) Модуль задач и дедлайнов
- [ ] Реализовать CRUD учебных задач (ДЗ, лабораторная, курсовая и т.д.).
- [ ] Реализовать команды просмотра задач на сегодня/неделю.
- [ ] Реализовать приоритеты, дедлайны и статусы выполнения задач.
- [ ] Добавить фильтрацию и сортировку задач по срочности.

## 7) Напоминания и асинхронная обработка
- [ ] Реализовать планирование напоминаний по занятиям, экзаменам и задачам.
- [ ] Реализовать публикацию задач уведомлений в RabbitMQ.
- [ ] Реализовать `NotificationModule` (консьюмер очереди и отправка в Telegram).
- [ ] Добавить ретраи и dead-letter-политику для неуспешных уведомлений.

## 8) LLM-советник
- [ ] Реализовать команду обращения к советнику и сбор пользовательского контекста.
- [ ] Подключить внешний LLM API через отдельный адаптер.
- [ ] Реализовать шаблон промпта (расписание, задачи, экзамены, дедлайны, свободные окна).
- [ ] Добавить fallback-поведение при недоступности LLM API и понятное сообщение пользователю.

## 9) Календарные интеграции
- [ ] Реализовать адаптер Google Calendar API и/или Yandex Calendar API.
- [ ] Реализовать двустороннюю синхронизацию событий расписания и дедлайнов.
- [ ] Реализовать хранение соответствий `local_event_id <-> external_event_id`.
- [ ] Добавить стратегию устойчивости при недоступности внешнего календаря (локальное сохранение + уведомление пользователя).

## 10) Healthcheck и эксплуатация
- [ ] Реализовать endpoint `/healthcheck` без admin-авторизации.
- [ ] Добавить проверки доступности MongoDB, RabbitMQ и внешних интеграций (degraded status).
- [ ] Подготовить docker-compose для локального запуска всех зависимостей.
- [ ] Добавить централизованное логирование и кореляционные `request_id`.

## 11) Нефункциональные требования
- [ ] Обеспечить время ответа простых команд до 500 мс (без внешних API).
- [ ] Обеспечить импорт файла до 1000 событий за время до 10 секунд.
- [ ] Добавить retry-политику: обычные API — 1 сек, LLM API — до 30 сек.
- [ ] Реализовать ограничение частоты запросов и защиту от перегрузки.
- [ ] Добавить аудит безопасности хранения ключей и секретов.

## 12) Тестирование
- [ ] Unit-тесты для бизнес-модулей и парсеров.
- [ ] Integration-тесты для MongoDB/RabbitMQ/внешних адаптеров (моки/тест-контейнеры).
- [ ] E2E-сценарии по ключевым пользовательским потокам (`/start`, импорт, `/today`, напоминания).
- [ ] Нагрузочные тесты для команд `/menu`, `/today`, `/tasks_today`.

## 13) Документация
- [ ] Обновить `README`: запуск, переменные окружения, команды бота, примеры импорта.
- [ ] Подготовить OpenAPI/Swagger документацию административного API.
- [ ] Описать схемы взаимодействия: Telegram API, LLM API, Calendar API, RabbitMQ.
- [ ] Подготовить глоссарий терминов и ограничения системы.

- [ ] `/start`, `/menu`, `/today`, `/tomorrow`, `/week`.
- [ ] Импорт CSV/iCal.

- [ ] `/healthcheck`.

- [ ] Расширенная аналитика и приоритизация задач.

### Release 3
